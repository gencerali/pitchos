# Kartalix — Reconciliation & Forward-Plan Prompt

## Context

The founder (Ali) had a long strategic conversation with another
Claude instance and produced 11+ proposal documents in `temp/`
without grounding them in the actual project state.

The project already has mature documentation discipline:
- `docs/ROADMAP.md` — releases v0.x → v1.0 → v1.x with explicit
  scope per release and parallel workstreams
- `docs/NEXT.md` — single-next-action discipline, updated at end
  of every session
- `docs/DECISIONS.md` — append-only architectural decision log
- `docs/SLICES.md` — sprint inventory (Sprints H done; I done;
  J/K/L specified)
- `docs/ARCHITECTURE-AUDIT-2026-05-15.md` — existing audit with
  P0/P1/P2 findings
- `docs/sprint-k-analysis.md` and `docs/sprint-l-analysis.md` —
  already contain pre-development analysis of the situational
  awareness and alarm framework briefs

Many of the founder's recent proposals (in `temp/`) duplicate or
contradict work already specified in these files. This prompt
asks you to reconcile and produce ONE clean forward-plan output.

## Your task

Do this in one focused session. Produce ONE markdown document at
the end. Do not start coding.

### Step 2 — Produce ONE reconciliation document

Save to `docs/RECONCILIATION-2026-05-17.md` with this exact
structure:

```
# Reconciliation Report — 2026-05-17

## 1. Project State Snapshot
- Current release: [from ROADMAP.md]
- Current NEXT action: [from NEXT.md, one line]
- Active sprints: [from SLICES.md "CURRENTLY IN FLIGHT"]
- Most recent decisions: [3 most recent from DECISIONS.md]
- Open P0/P1 from architecture audit: [count + one-liner each]

## 2. Brief-by-Brief Reconciliation
For each file in temp/, one block:

### temp/[filename]
- **Topic**: [one line]
- **Status in current docs**:
  - DUPLICATE: already covered by [ROADMAP entry / SLICE / NEXT
    line / DECISION]
  - REFINEMENT: adds detail to existing [thing] without conflict
  - CONFLICT: contradicts [thing] in [file]
  - NEW: not yet covered anywhere
  - OBSOLETE: was relevant when written but is now overtaken
    by [thing]
- **Recommended disposition**: KEEP / MERGE INTO [doc] / ARCHIVE
  / DELETE
- **One-sentence rationale**

## 3. Items Already Done That The Briefs Treat As Open
Walk through and list every brief item that is already done in
NEXT.md or implemented in code. Be specific. Each entry:
- Brief item: [what the brief proposed]
- Already done: [where/when, with date]
- Action: archive that section of the brief

## 4. Items in Briefs That Should Become Roadmap Entries
For each genuinely NEW item, write the proposed ROADMAP entry:
- Where it belongs (which release section)
- One-paragraph scope
- Dependencies
- Estimated effort (S/M/L/XL)
- Why it matters for v1.0 vs post-launch

For each strategic decision implied in the briefs but not yet in

- e.g. "No multi-step LLM agents — single-prompt LLM calls and
  scheduled functions only"
- e.g. "Strangler over big-bang rebuild for content-shape
  refactor"
- e.g. "AI disclosure is positive for AdSense in 2026"

## 6. Open Questions That Block Forward Progress
The brief inventory listed 8 open questions. For each, either:
- Answer it now from code/docs inspection
- Confirm it remains open and add to NEXT.md as a discovery task

Specifically check:
- Are rss_summary articles currently served at public /haber/*
  URLs? (read serveArticlePage and renderArticleHTML)
- Does sitemap include rss_summary articles?
- Does article HTML include a byline? What does it say?

- Does each cron run write a success/failure record somewhere?
- What % of published articles in last 30 days have a story_id?
  (Supabase query)
- Are any /force-* endpoints still missing requireOps? (grep the
  worker)
- What is the current daily Claude API spend?
  (admin/cost data)

## 7. Recommended Forward Plan (Next 4 Weeks)
Concrete week-by-week plan that:
- Respects the existing release cadence (v0.96 → v0.97 → v0.98
  → v1.0)
- Adds the new items from Section 4 into the right releases
  (don't invent a new release)

  alongside Sprint J or K

Format as:
- Week of [date]: [headline]. NEXT action: [one line for NEXT.md]
- ...

## 8. Files to Archive
Anything in temp/ that should be moved to docs/legacy/ (kept for
reference) or deleted. Be specific about each file.

## 9. Files to Update
List every existing doc file that needs an update as a result of

- docs/NEXT.md: [change]
- docs/ROADMAP.md: [change]
- docs/DECISIONS.md: [new entry to append]
- docs/SLICES.md: [change]

## 10. What I Did Not Do
Be explicit about anything you couldn't resolve and why. Do not
silently skip.
```
